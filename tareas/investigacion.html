Tarea 03 — Parte Teórica

Elementos utilizados para formularios en HTML — (contenido íntegro del documento, sin carátula).

Índice

    Introducción

    Los formularios HTML son componentes fundamentales de los sitios web interactivos. Permiten recopilar información de los usuarios mediante campos de entrada, selecciones y botones para posteriormente procesarla (por ejemplo, enviarla a un servidor). Un formulario típico combina elementos HTML (como campos de texto, casillas, menús, etc.) con estilos CSS para mejorar su apariencia y usabilidad, y puede apoyarse en frameworks de CSS para agilizar su diseño. Además, gracias a técnicas de diseño adaptable como las media queryes, es posible lograr que los formularios se vean y funcionen correctamente en diferentes tamaños de pantalla (móvil, tableta, escritorio). La accesibilidad asegura que todos los usuarios (incluyendo personas con discapacidades) puedan interactuar con los formularios sin barreras, mientras que la validación (tanto con funciones nativas de HTML5 como mediante JavaScript) garantiza la integridad y corrección de los datos ingresados.

    En este informe se investigan en profundidad los elementos HTML más utilizados en formularios (especialmente <input>, <textarea> y <select>), la aplicación de CSS y pseudoclases para dar estilo y retroalimentación visual a los campos, el uso de frameworks de CSS (en particular Bootstrap y Tailwind CSS) para facilitar el desarrollo de formularios modernos, el empleo de media queryes para lograr diseños responsivos, y las mejores prácticas de accesibilidad y validación de formularios. Todo ello se ilustrará con ejemplos de código bien formateados, imágenes demostrativas de formularios terminados y referencias académicas y técnicas que respaldan las buenas prácticas presentadas.

    El elemento <input> y sus tipos

    El elemento HTML <input> define un control de entrada de datos versátil y configurable dentro de un formulario. Mediante su atributo obligatorio type, un <input> puede comportarse de múltiples formas para aceptar diferentes tipos de datos o proporcionar diversos widgets de entrada. A continuación se describen los principales tipos de <input> disponibles en HTML5, junto con sus usos y características más importantes:

    text: campo de texto de una sola línea para entrada libre de texto plano. Es el tipo por defecto (si no se especifica type) y se utiliza para nombres, búsquedas u otras entradas de texto genérico.

    password: campo de texto que oculta los caracteres ingresados (sustituyéndolos por puntos o asteriscos) para proteger información sensible como contraseñas. Su valor no se muestra en pantalla al teclear.

    email: campo de texto especialmente validado para admitir solo direcciones de correo electrónico con formato válido. Los navegadores realizan una comprobación automática del patrón del email e incluso pueden mostrar un teclado optimizado en dispositivos móviles. Si el contenido no coincide con el formato de un email válido, el campo se considera inválido (:invalid).

    number: campo que acepta únicamente números (enteros o decimales). Suele mostrarse con flechas de incremento/decremento en navegadores de escritorio. Puede usar atributos como min, max y step para delimitar el rango numérico permitido.

    tel: campo para introducción de números de teléfono. En móviles suele disparar un teclado numérico. No aplica formateo o validación automática más allá de ser texto sin restricciones (difiere de number en que permite caracteres como + o - útiles en teléfonos).

    url: campo validado para direcciones web (URLs). De forma similar al email, el navegador verifica que el texto ingresado tenga formato de URL (por ejemplo, que comience con http:// o https://) y aplica pseudoclases de válido/inválido según corresponda.

    date / time / datetime-local: controles para selección de fechas y horas. Por ejemplo, date muestra un selector de fecha (año, mes, día) apropiado para la localidad del usuario; time permite escoger una hora (HH:MM:SS); datetime-local combina fecha y hora en un solo control. Facilitan que el usuario introduzca datos temporales válidos mediante interfaces nativas del navegador (como calendarios emergentes).

    color: selector de color. Suele mostrarse un pequeño cuadro que al hacer clic despliega una paleta o picker de color para elegir un valor (normalmente produce un código hexadecimal del color elegido).

    file: campo para carga de archivos. Al hacer clic en él, abre el explorador de archivos del sistema para seleccionar uno o varios archivos (según el atributo multiple). Permite restringir qué tipos de archivo se pueden seleccionar mediante el atributo accept (por ejemplo, accept="image/png, image/jpeg" limita a imágenes PNG o JPEG).

    checkbox: casilla de verificación (checkbox) que representa una opción booleana (verdadero/falso). Se usa típicamente en grupos donde múltiples opciones pueden estar seleccionadas a la vez (marcar/desmarcar independientes). Cada checkbox debería llevar un atributo value (valor enviado si está marcado) y opcionalmente checked para indicar que está marcado por defecto.

    radio: botón de opción (radio button) utilizado en grupos excluyentes. Varias opciones de tipo radio que comparten el mismo nombre de grupo (name) permiten seleccionar solo una de ellas a la vez. Al igual que checkbox, puede tener checked para predeterminar la selección inicial.

    range: control para seleccionar un valor de un rango numérico mediante un slider (deslizador). El usuario puede arrastrar una perilla horizontalmente para elegir un número dentro de un rango continuo. Debe usarse con atributos min, max y step para definir el intervalo y granularidad.

    hidden: campo oculto que no se muestra al usuario pero cuyo valor se envía con el formulario. Útil para incluir metadatos, tokens u otra información que no requiere interacción del usuario.

    submit: botón de envío del formulario. Al hacer clic (o pulsar Enter en algún campo de texto), desencadena la acción del formulario (<form action="..." method="...">). Suele mostrar texto "Enviar", "Submit" u otro indicativo.

    reset: botón que reinicia todos los campos del formulario a sus valores iniciales (o vacíos si no tenían valor por defecto). Se usa con precaución pues su utilidad es limitada y puede confundir a usuarios, por lo que se suele evitar.

    button: botón genérico sin comportamiento predeterminado (no envía el formulario). Se utiliza junto con JavaScript para acciones personalizadas (por ejemplo, un botón que ejecuta cierta lógica al hacer clic). Alternativamente, en vez de <input type="button"> se suele preferir <button> (elemento separado) por su flexibilidad.

    Cada tipo de <input> admite atributos específicos que enriquecen su comportamiento. Por ejemplo, required marca un campo como obligatorio (el formulario no se podrá enviar si está vacío), disabled lo deshabilita (no editable/grisado), maxlength y minlength definen longitudes máximas/mínimas de texto, pattern permite exigir que el texto coincida con una expresión regular, placeholder muestra un texto de sugerencia cuando el campo está vacío, entre otros. Estas restricciones nativas de HTML5 proporcionan una primera capa de validación de datos en el lado del cliente de manera sencilla.

    Ejemplo de código

    A continuación se muestra un fragmento de HTML con varios campos <input> de distintos tipos dentro de un formulario, ilustrando su sintaxis de uso común (con comentarios explicativos):

    <form action="/registro" method="post">

  <!-- Campo de texto libre -->
  <label>Nombre:
    <input type="text" name="nombre" placeholder="Escribe tu nombre" required>
  </label>

  <!-- Campo de contraseña -->
  <label>Contraseña:
    <input type="password" name="clave" required>
  </label>

  <!-- Campo de email con validación nativa -->
  <label>Email:
    <input type="email" name="correo" placeholder="usuario@ejemplo.com">
  </label>

  <!-- Campo numérico con rango definido -->
  <label>Edad:
    <input type="number" name="edad" min="1" max="120">
  </label>

  <!-- Casilla de verificación -->
  <label>
    <input type="checkbox" name="subscripcion" value="newsletter">
    Suscribirse al boletín
  </label>

  <!-- Botones de radio agrupados (mismo name) -->
  <div>
    Sexo:
    <label><input type="radio" name="sexo" value="M"> Masculino</label>
    <label><input type="radio" name="sexo" value="F"> Femenino</label>
  </div>

  <!-- Botón de envío del formulario -->
  <input type="submit" value="Registrar">

</form>

    En el ejemplo anterior, se utiliza un elemento <form> con distintos <input> dentro. Las etiquetas <label> envuelven o se asocian a cada control para que al hacer clic en el texto descriptivo se active el campo correspondiente (mejora de usabilidad y accesibilidad). Se incluyen algunos atributos como placeholder (ej. en el email) que proporciona una pista al usuario, required en campos obligatorios, y atributos de rango (min, max) para limitar valores numéricos. Al final, un <input type="submit"> actúa como botón que envía los datos. Esta variedad cubre los casos de uso más comunes de entradas en formularios.

    El elemento <textarea>

    El elemento HTML <textarea> representa un campo de texto de entrada multilínea, es decir, un cuadro de texto de área grande donde el usuario puede escribir párrafos o textos de extensión considerable (múltiples líneas, con saltos de línea). A diferencia de <input type="text"> que solo permite una línea de texto, <textarea> ofrece un cuadro expandible verticalmente para contenidos más largos, como comentarios, descripciones, mensajes, etc..

    La sintaxis básica de <textarea> requiere una etiqueta de apertura y otra de cierre (no es un elemento vacío). Entre dichas etiquetas se puede colocar texto que sirva como valor inicial o por defecto del campo (el usuario lo verá al cargar la página y podrá modificarlo). Ejemplo mínimo:

    <label for="coment">Comentario:</label>
<textarea id="coment" name="comentarios" rows="4" cols="50" placeholder="Escribe aquí tu comentario..."></textarea>

    En este ejemplo, rows="4" y cols="50" son atributos que establecen un tamaño inicial del área de texto (4 líneas de alto por 50 caracteres de ancho). Sin embargo, estos atributos son puramente iniciales; hoy en día es más común controlar el tamaño de <textarea> mediante CSS (propiedades width, height u otras) y permitir que sea redimensionable libremente por el usuario (muchos navegadores permiten agarrar la esquina inferior derecha y ajustar el tamaño).

    Algunos atributos importantes de <textarea> incluyen:

    • name: nombre del campo (clave con la que se enviará el contenido en el formulario).
    • rows y cols: tamaño visible en líneas y columnas.
    • placeholder: texto de sugerencia cuando el campo está vacío.
    • maxlength / minlength: límites de longitud.
    • required: indica que no puede quedar vacío.
    • readonly / disabled: solo lectura / deshabilitado.
    • wrap: control de ajuste de líneas (“soft” o “hard”).

    Un detalle importante es que <textarea> soporta la propiedad CSS resize para permitir o no que el usuario cambie su tamaño manualmente. En diseño web, a veces se restringe a solo vertical (resize: vertical) para mantener el ancho consistente.

    Accesibilidad: Siempre asociar un <label> claramente, usando for/id o envolviendo el control. Puede añadirse ARIA cuando haga falta.

    Ejemplo de <textarea> con estilos

    <style>
  textarea.feedback:focus {
    border-color: #66afe9; /* borde azul al enfocar */
    outline: none;         /* quitar el contorno por defecto */
    box-shadow: 0 0 5px rgba(102,175,233,0.5);
  }
</style>

<label for="fb">Tu opinión:</label><br>
<textarea id="fb" name="feedback" class="feedback" rows="5" cols="60"
  maxlength="300" placeholder="Escribe tu opinión (máx. 300 caracteres)" required></textarea>
<span id="contador">300 caracteres disponibles</span>

    En este fragmento, la clase .feedback aplicada al <textarea> resalta el foco. Además, el atributo maxlength="300" limita la entrada y se podría usar JavaScript para actualizar el contador. En resumen, <textarea> es esencial cuando necesitamos entradas de texto largas.

    El elemento <select>

    El elemento <select> define un menú desplegable de opciones en HTML. Dentro de <select> se anidan elementos <option>, cada uno representando una de las opciones que el usuario puede elegir. También es posible agrupar opciones en categorías mediante <optgroup> si es necesario.

    Ejemplo básico

    <label for="pais">País:</label>
<select name="pais" id="pais" required>
  <option value="">--Seleccione--</option>
  <option value="mx">México</option>
  <option value="es">España</option>
  <option value="ar">Argentina</option>
  <optgroup label="Otros">
    <option value="ot">Otro país</option>
  </optgroup>
</select>

    Atributos principales de <select>: name, disabled, required, multiple, size. En <option>: value, selected.

    Selección múltiple

    <label for="asignaturas">Selecciona tus asignaturas favoritas:</label><br>
<select id="asignaturas" name="asig_fav[]" multiple size="5">
  <option value="mat">Matemáticas</option>
  <option value="fis" selected>Física</option>
  <option value="qui">Química</option>
  <option value="his" selected>Historia</option>
  <option value="lit">Literatura</option>
  <option value="art">Arte</option>
</select>

    Aquí multiple permite elegir varias materias; el nombre con [] suele interpretarse como arreglo en el servidor.

    CSS y estilo de formularios

    El lenguaje CSS (Cascading Style Sheets) se usa para definir la presentación visual de los documentos HTML, incluyendo los formularios. A través de CSS, podemos modificar la apariencia de los campos de formulario (inputs, textareas, selects, botones, labels, etc.) para hacerlos más atractivos, claros y usables. Algunas de las propiedades de CSS más utilizadas en formularios son:

    Colores y fondos: mediante color, background-color se puede resaltar campos obligatorios, cambiar el color de botones, etc. También border, border-radius y box-shadow para bordes/curvas/sombras.

    Espaciado: margin y padding para separar campos y dar comodidad visual.

    Tipografía: font-size, font-family, font-weight para legibilidad.

    Layout (Diseño): Flexbox y Grid para disposición de etiquetas/campos y columnas responsivas.

    Grid de dos columnas (labels/inputs)

    .form-grid {
  display: grid;
  grid-template-columns: 1fr 2fr; /* columna estrecha para labels, ancha para inputs */
  gap: 1em;
}
.form-grid label { text-align: right; padding-right: 0.5em; }
    <div class="form-grid">
  <label for="user">Usuario:</label>
  <input id="user" type="text" name="user">
  <label for="email">Email:</label>
  <input id="email" type="email" name="email">
</div>

    Estados visuales con pseudoclases

    input:invalid { border-color: #e74c3c; }
input:valid   { border-color: #2ecc71; }

    Se recomienda activar estilos de error tras intento de envío para no marcar en rojo desde el inicio. Los navegadores aplican estilos por defecto; puede usarse reset/normalize o Reboot (Bootstrap).

    Frameworks de estilo (Bootstrap y Tailwind CSS)

    Bootstrap

    Ofrece componentes y grid responsive. En formularios: .form-control, rejilla (.row, .col-*), .btn, y validación visual con .was-validated, .is-valid/.is-invalid y .invalid-feedback.

    <form class="needs-validation" novalidate>
  <div class="mb-3">
    <label for="email" class="form-label">Correo electrónico</label>
    <input type="email" class="form-control" id="email" required>
    <div class="invalid-feedback">Ingrese un email válido.</div>
  </div>
  <div class="mb-3">
    <label for="pass" class="form-label">Contraseña</label>
    <input type="password" class="form-control" id="pass" minlength="6" required>
    <div class="invalid-feedback">Mínimo 6 caracteres.</div>
  </div>
  <div class="mb-3 form-check">
    <input type="checkbox" class="form-check-input" id="terms" required>
    <label class="form-check-label" for="terms">Acepto términos</label>
    <div class="invalid-feedback">Debe aceptar antes de enviar.</div>
  </div>
  <button type="submit" class="btn btn-primary">Registrarse</button>
</form>

    Tailwind CSS

    Enfoque utilitario: compones el diseño con clases pequeñas; recomiendan el plugin @tailwindcss/forms para controles base.

    <div class="max-w-md bg-white p-6 rounded shadow">
  <form>
    <div class="mb-4">
      <label class="block text-gray-700 font-semibold mb-2" for="email">Email</label>
      <input class="block w-full border border-gray-300 rounded py-2 px-3
                     focus:border-blue-500 focus:outline-none"
             type="email" id="email" placeholder="correo@ejemplo.com">
    </div>

      <label class="block text-gray-700 font-semibold mb-2" for="msg">Mensaje</label>

      Enviar

</div>

    Comparación Bootstrap vs Tailwind

    • Componentes vs utilidades: rapidez estándar vs control granular.
    • Personalización: Sass/overrides vs configuración total + purga.
    • Curva: Bootstrap más directa; Tailwind exige base sólida de CSS.
    • Tamaño: Tailwind purga clases no usadas; Bootstrap incluye estilos de componentes.
    • HTML vs CSS: en Tailwind casi todo va en el HTML.

    Media Queries y diseño responsivo

    El concepto de diseño responsivo consiste en crear páginas web (y formularios) que se adapten fluidamente a distintos tamaños de pantalla. En CSS, la herramienta fundamental son las Media Queries.

    @media (max-width: 768px) {
  /* Reglas solo si el viewport tiene ancho <= 768px */
  form input, form select, form textarea {
    width: 100%;
  }
}

    Puntos de quiebre típicos: 576px, 768px, 992px. Estrategias: reorganizar columnas, ocultar/reducir ayudas en móvil, ajustar tipografías/espacios, y cambiar disposición de botones. Se recomienda enfoque mobile first (min-width).

    Accesibilidad y validación en formularios

    Buenas prácticas:

    • <label> obligatorio: asociar por for/id o envolviendo el control; no depender solo de placeholder.
    • ARIA cuando sea necesario: aria-label, aria-labelledby, aria-describedby, aria-invalid.
    • Mensajes de error claros: específicos, visibles, asociados al campo y no depender solo del color.
    • Navegación por teclado: orden lógico; evitar tabindex positivos; usar fieldset/legend para agrupar.
    • Controles táctiles adecuados: blancos ~44×44px; <label> clicable para check/radio.
    • Autocomplete: usar valores adecuados (ej. email, new-password).
    • Validación: HTML5 + JS; siempre validar en servidor.

    Resumen: usar semántica nativa, añadir ARIA solo si hace falta, errores amables y oportunos, y probar con teclado/lectores de pantalla.

    Conclusión

    El desarrollo de formularios web efectivos involucra una combinación de conocimiento técnico y buenas prácticas de experiencia de usuario. En este informe hemos explorado los aspectos fundamentales para crear formularios que no solo luzcan profesionales, sino que también sean funcionales, accesibles y adaptables.

    Conocer los elementos HTML base (<input>, <textarea>, <select>), aplicar CSS moderno (Flex/Grid y pseudoclases), elegir un framework (Bootstrap o Tailwind) según contexto, usar media queries para un diseño responsive y cuidar accesibilidad/validación, son pilares para formularios de alta calidad.
